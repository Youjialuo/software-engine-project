## 1. 敏捷的本质——从"按计划执行"到"按反馈调整"

### 1.1 敏捷方法的核心定义

课件开门见山地定义了敏捷:

> *Methode de gestion de processus de developpement incrementale.*
> 敏捷是一种**增量式**的过程管理方法。

关键特征:
1. **快速推向市场**: 产品必须尽快上市 (les produits doivent etre dans le marche sans tarder),因此实现和部署必须快速
2. **开发与测试并发**: 开发活动和测试活动**同时进行** (concurrentes)
3. **面对面沟通**: 鼓励团队合作和面对面交流
4. **主流地位**: 今天,绝大多数软件都使用敏捷方法开发

### 1.2 敏捷思维的三个层次

课件引用了敏捷思维模型,从外到内分为三层:

```
外环 (Practices):   Scrum, XP, Kanban  —— 做什么 (具体实践)
中环 (Principles):  敏捷12条原则          —— 如何思考
内环 (Values):      敏捷4大价值观         —— 为什么 (信念层)
核心 (Mindset):     敏捷心态              —— 最深层的文化转变
```

**深层解读**: 许多组织"做敏捷"失败的根源在于只在最外层模仿实践(如站会、Scrum板、Sprint),但没有改变中层和内核的价值观和心态。这就是所谓的"僵尸Scrum"(Zombie Scrum)——有Scrum的形式,没有Scrum的灵魂。

### 1.3 敏捷为何诞生?

传统的瀑布模型假设需求可以在项目初期被完整捕获并冻结。但现实是:
- 用户在看到可运行的软件之前,不知道自己真正需要什么
- 市场变化速度远超传统开发周期
- 软件的核心价值不在于"符合规格说明",而在于"解决用户问题"

敏捷正是对这些痛点的回应: **接受变化,拥抱变化,利用变化获取竞争优势**。

## 2. 敏捷宣言——四大价值观与十二条原则

### 2.1 四大价值观 / Les 4 Valeurs Agile

课件列出了敏捷宣言的核心——注意格式的特殊性: 虽然重视右边的项目,但**更重视左边的项目**。

| # | 我们重视... | 胜过... |
|---|------------|---------|
| 1 | **个体与互动** (Individus et interactions) | 流程与工具 (Processus et outils) |
| 2 | **可工作的软件** (Logiciel operationnel) | 详尽的文档 (Documentation exhaustive) |
| 3 | **与客户合作** (Collaboration avec le client) | 合同谈判 (Negociation contractuelle) |
| 4 | **响应变化** (Adaptation au changement) | 遵循计划 (Suivi d'un plan) |

**深层解读**:
- 价值观1说明: 工具和流程有用,但当它们阻碍人与人的协作时,应该被简化或绕过。
- 价值观2说明: 文档有价值,但如果同一时间可以用来写可工作的代码,优先写代码。这**不是**说不要文档,而是文档要适度。
- 价值观3说明: 对抗性合同关系(甲方vs乙方)让双方关注"如何保护自己"而非"如何一起成功"。合作型关系让双方关注共同目标。
- 价值观4说明: 计划必不可少,但计划应该被视为**假设**而非承诺。当现实与假设不符时,调整计划而非固守。

### 2.2 十二条原则 / Les 12 Principes Agile

课件原文逐条列出了敏捷12原则(完整引用自敏捷宣言):

| # | 原则 (核心含义) | 深层解读 |
|---|---------------|---------|
| 1 | **客户满意度优先**: 通过快速持续交付有价值的功能满足客户 | "有价值的功能"是关键——不是任何功能,而是用户真的需要的 |
| 2 | **欢迎需求变更**: 即使晚期,利用变更为客户创造竞争优势 | 这在传统模型中是不可想象的——变更 = 成本;在敏捷中,变更 = 机会 |
| 3 | **频繁交付**: 数周到数月的周期,越短越好 | 短周期 = 快速反馈 = 快速纠偏 |
| 4 | **业务与开发协作**: 用户/代表与开发者每天一起工作 | 消除"翻译误差"——业务方直接对开发者表达需求 |
| 5 | **激励个体**: 提供环境和支持,信任他们完成任务 | 管理者的角色从"控制者"变为"赋能者" |
| 6 | **面对面沟通**: 最有效的传递信息方式 | 文字有歧义,对话可以追问;团队应尽可能同地办公 |
| 7 | **可工作的软件是首要度量标准** | 不是代码行数、工时、故事点数——是可运行的软件 |
| 8 | **可持续节奏**: 无限期保持恒定的开发节奏 | 反对"死亡行军"(Death March)——长期加班会摧毁生产力 |
| 9 | **技术卓越**: 持续关注技术卓越和良好设计 | 敏捷 ≠ 快速且肮脏;真正的敏捷需要高质量代码支持快速变化 |
| 10 | **简单**: 最大化未完成工作量的艺术 | KISS原则;YAGNI(You Aren't Gonna Need It) |
| 11 | **自组织团队**: 最好的架构、需求和设计出自自组织团队 | 自组织 ≠ 无组织;团队自己决定"如何做"而非被命令 |
| 12 | **定期反思与调整**: 团队定期反思如何变得更有效,然后调整 | 这是元认知层面的敏捷——"敏捷的敏捷": 不断改进敏捷实践本身 |

**关键关联**: 原则9(技术卓越)与传统认知"敏捷就是快速写代码"相矛盾。实际上,真正的敏捷**必须**以技术卓越为前提,因为只有高质量的代码才能承载快速和频繁的变化。没有技术卓越的"敏捷"最终会陷入技术债务的泥潭。

## 3. 用户故事 (User Stories)——需求的最小表达单元

### 3.1 定义与目的

> 课件定义: *Une phrase simple decrivant une fonctionnalite logicielle, redigee du point de vue de l'utilisateur final.*
> 一个简单的句子,从最终用户的角度描述软件功能。

**目的**: 精确地表示一个功能特性为最终用户带来的**附加值**(valeur ajoutee)——即这个功能对用户产生的实际影响是什么。

### 3.2 标准格式 (Template)

```
En tant que [profil client], je souhaite [objectif du logiciel] afin de [resultat].
作为 [用户类型], 我想要 [软件目标] 以便 [获得结果].
```

**实例**:
- "作为**在线购物者**,我想要**按价格筛选商品**,以便**在我的预算范围内找到合适的产品**"
- "作为**系统管理员**,我想要**查看用户登录日志**,以便**审计可疑活动**"

### 3.3 3C原则 (补充知识)

课件未展开但标准Scrum教材包含的3C:
- **Card**(卡片): 故事写在一张物理/虚拟卡片上,提醒需要进行对话
- **Conversation**(对话): 故事是对话的占位符,详情在开发者和PO之间讨论
- **Confirmation**(确认): 验收标准定义"完成"意味着什么

### 3.4 用户故事驱动验收测试

课件指出: 用户故事**驱动验收测试**(pilote les tests d'acceptation)。这意味着每个故事的"完成"标准不是"代码写完了",而是"通过了对应的验收测试"。

### 3.5 量化参考

> 课件数据: **80 +/- 20 个用户故事 = 一个新版本**

这意味着一个典型的发布版本包含60-100个用户故事,为工作量估算提供了粗略的基准。

## 4. 极限编程 (XP)——纪律化的敏捷实践

### 4.1 XP的五大价值观

课件列出XP建立在五个核心价值观之上:

| 价值观 | 法文 | 深层含义 |
|--------|------|---------|
| **沟通** | Communication | 频繁且高效的沟通,让所有参与方理解项目目标、客户需求和开发步骤 |
| **反馈** | Feedbacks | 让开发者实时调整,根据客户的实际需求和意见修改正在设计中的交付物 |
| **勇气** | Courage | 客户需求是项目中心;开发者必须准备好多次推翻数周的工作 |
| **尊重** | Respect | 润滑沟通,确保团队成员之间相互尊重 |
| **简单** | Simplicite | 优先选择最显而易见的解决方案 |

### 4.2 XP的12项核心实践

课件详细列出了XP的核心工程实践:

| # | 实践 | 法文 | 详解 |
|---|------|------|------|
| 1 | **测试驱动开发** | Developpement pilote par les tests | 先写测试,后写实现。迭代完成标准: 代码通过本迭代及所有前序测试 |
| 2 | **计划游戏** | Jeu du planning | 客户写用户故事 -> 团队评估时间(通常用Fibonacci) -> 客户按优先级和时间选择本迭代的故事 |
| 3 | **结对编程** | Programmation en binome | 两人共用一个键盘。驾驶员(Pilote)写代码,导航员(Copilote)审查、建议。频繁交换角色和搭档 |
| 4 | **持续集成** | Integration en temps reel | 功能一完成就立即集成到产品,避免后期大规模集成的痛苦 |
| 5 | **简单设计** | Conception simple | 只设计当前需要的,不过度设计 |
| 6 | **持续重构** | Refactoring continu | 不断消除重复和冗余,保持代码简洁可读、易于修改 |
| 7 | **编码标准** | Uniformisation du code | 统一术语和约定,使代码易于阅读和修改 |
| 8 | **集体代码所有权** | - | 任何人对任何代码都可以修改 |
| 9 | **小型发布** | - | 尽可能小的、频繁的发布 |
| 10 | **隐喻** | Utilisation de metaphores | 用最简方式解释项目全局,确保团队整体理解 |
| 11 | **可持续节奏** | - | 不过度加班 |
| 12 | **现场客户** | - | 客户在场随时回答问题 |

### 4.3 XP四象限

课件还从四个维度定义了XP关注的核心:

```
     Risque (风险)
        |
Coût ----+---- Delai (期限)
 (成本)   |
     Qualite (质量)
```

**XP的核心论点**: 如果你想控制风险、成本、期限和质量,你需要沟通、反馈、勇气、尊重和简单。XP不是"没有过程",而是一种**极度纪律化的工程实践集**。

## 5. Scrum——最广泛使用的敏捷框架

### 5.1 Scrum的本质

> *Scrum est la plus utilisee des methodologies agiles.*
> Scrum是使用最广泛的敏捷方法论。

**三大支柱**(在后续课件幻灯片中补充):
1. **透明性**(Transparence): 流程的各个方面对所有责任人可见
2. **检查**(Inspection): 频繁检查工件和进展以发现偏差
3. **适应**(Adaptation): 如果检查发现偏差超出可接受范围,调整流程或材料

### 5.2 经验主义方法论

课件强调: Scrum是**经验主义方法**(Methode empirique)——基于观察和实验。决策是基于"此时此刻可用的信息"做出的,而非基于预先制定的长期计划。

这与传统"定义式过程控制"(Defined Process Control)形成对比:
- **定义式**: 给定输入,我精确知道输出;适用于高度可预测的领域(如制造业)
- **经验式**: 输入和输出之间的关系不完全已知;适用于复杂和不可预测的领域(如软件开发)

### 5.3 时间盒机制

课件强调Scrum中的**时间盒**(Delimitations dans le temps):

| 时间盒 | 典型时长 | 说明 |
|--------|---------|------|
| **Sprint** | 2-4周 | 一个完整的开发周期 |
| **Daily Scrum** | 15分钟 | 每日站会 |
| **Sprint Planning** | Sprint时长相关 | Sprint计划会议 |
| **Sprint Review** | Sprint时长相关 | Sprint评审 |
| **Sprint Retrospective** | Sprint时长相关 | Sprint回顾 |

## 6. Scrum三大角色

### 6.1 角色全景

课件用较大篇幅详细讲解了Scrum的五个角色(3个Scrum核心角色 + 2个组织角色):

```
Product Owner (产品负责人) -----> 价值最大化
       |
Scrum Master (敏捷教练) --------> 流程保护者
       |
Equipe de Developpement ---------> 交付产品
 (开发团队, 5-9人)
```

### 6.2 Product Owner (产品负责人)

**职能**: 业务/客户的声音 (voix de l'entreprise/du client)

| 职责 | 详解 |
|------|------|
| **从业务角度驱动项目** | 非技术管理,而是价值管理 |
| **传达清晰的愿景** | 定义产品特性,确保团队理解"为什么" |
| **接受或拒绝交付** | 每个Sprint结束时正式验收 |
| **确保团队聚焦最高价值** | 管理Product Backlog的优先级排序 |
| **负责ROI和交付** | 投资回报率和交付节奏 |

### 6.3 Scrum Master (敏捷教练)

**职能**: 流程的守护者,团队的盾牌。

| 核心职责 | 详解 |
|---------|------|
| **保护团队** | 免受外部干扰和"漩涡"(turbulences) |
| **清除障碍** | 排除阻碍团队进展的一切因素 |
| **执行流程** | 确保Scrum的规则被理解和遵守 |
| **辅导团队** | 优化生产力,推进"检查-适应"循环 |
| **促进组织变革** | 与Product Owner合作,改变组织以支持Scrum |

**关键澄清**: Scrum Master **不是**团队成员(不是开发团队的成员),**不负责**交付物。他们不写代码,不开会做决策——他们确保**流程**正确运行。

### 6.4 开发团队 (Equipe de Developpement)

**职能**: 交付产品并对质量负责。

课件的关键定义:

| 特性 | 说明 |
|------|------|
| **规模** | 5-9人 (经典"两个披萨团队"规模) |
| **多学科** (Multidisciplinaire) | 团队拥有交付增量所需的所有技能 |
| **自管理** (Autogeree) | 团队自己决定"如何做" |
| **跨职能** (Cross-fonctionnelle/Transverse) | 基于技能而非基于职能的团队构成 |
| **自组织** | 团队自己组织自己的工作 |

### 6.5 客户与管理者

| 角色 | 法文 | 职责 |
|------|------|------|
| **客户** | Client | 订购产品、付费、给予反馈和修订 |
| **管理者** | Manager | 确保组织在失败时存活、制定规则和方针 |

## 7. Scrum三大工件与四大事件

### 7.1 三大工件 (Artefacts)

| 工件 | 负责人 | 内容 |
|------|--------|------|
| **Product Backlog** | Product Owner | 所有需求的优先级排序列表;持续更新 |
| **Sprint Backlog** | 开发团队 | 当前Sprint要交付的需求子集 |
| **Increment** | 开发团队 | 每个Sprint结束时产出的可工作软件 |

### 7.2 四大事件 (Evenements)

| 事件 | 时长 | 参与者 | 目的 |
|------|------|--------|------|
| **Sprint Planning** | 最多8h/月Sprint | 全团队 | 定义Sprint目标,选择Backlog条目 |
| **Daily Scrum** | 15分钟 | 开发团队 | 同步进展,识别障碍 |
| **Sprint Review** | 最多4h/月Sprint | 全团队+干系人 | 展示增量,收集反馈 |
| **Sprint Retrospective** | 最多3h/月Sprint | 全团队 | 反思流程,制定改进计划 |

## 8. XP vs Scrum——互补而非竞争

### 8.1 核心区别

| 维度 | XP | Scrum |
|------|----|-------|
| **关注点** | 工程实践(怎么写代码) | 管理框架(怎么组织工作) |
| **实践范围** | 技术实践为主(TDD、结对编程、重构) | 流程实践为主(Sprint、站会、评审) |
| **角色定义** | 较宽松 | 明确定义PO/SM/Team |
| **适用层级** | 团队级技术方法 | 可扩展到组织级 |

### 8.2 互补关系

在实践中,XP和Scrum常结合使用: **Scrum管理"做什么"和"什么时候做",XP指导"怎么做"**。许多高绩效团队同时采用Scrum框架和XP技术实践。
