## 1. 架构模式的定义与架构师的核心关切

### 1.1 软件过程中的定位

课件首先将架构设计定位在软件过程的四大基本活动中:

```
需求规格 -> 设计与实现 -> 验证 -> 演进
(Specification)  (Conception)  (Validation)  (Evolution)
```

架构设计属于**设计(Conception)**阶段的核心活动。

### 1.2 架构模式的定义

> *Les Patrons sont un moyen de representer, de partager et de reutiliser les connaissances. Un Patron de conception (design pattern) est une description stylisee d'une bonne pratique de conception, qui a fait ses preuves dans differents environnements.*

架构模式 = 经过验证的、可复用的系统级设计决策模板。

### 1.3 架构师的六大核心问题

课件列出了架构设计师在系统设计时必须回答的问题:

| # | 问题 | 决定什么 |
|---|------|---------|
| 1 | 系统是**交互式**的吗? | 选择MVC还是其他 |
| 2 | 系统需要**频繁变更**吗? | 分层架构 vs 紧耦合 |
| 3 | 系统是否**分布在网络**上? | 是否采用Client-Server |
| 4 | 功能如何在组件间**分解**? | 关注点分离策略 |
| 5 | 是否有**通用架构**可用? | 复用已知模式 |
| 6 | 哪些**非功能性需求**最重要? | 性能/安全/可维护性的权衡 |

## 2. 架构的四大视图——4+1视图模型

### 2.1 为什么需要多视图?

课件指出: 每种架构模型只展示系统的**一个视角或透视图**。完整的架构文档需要多个视图互补。

### 2.2 四大视图

| 视图 | 法文 | 内容 | 回答的问题 |
|------|------|------|-----------|
| **逻辑视图** | Vue logique | 系统的关键抽象(对象/类) | "系统由哪些概念组成?" |
| **进程视图** | Vue des processus | 运行时的交互进程 | "系统如何并发执行?性能和可用性如何?" |
| **开发视图** | Vue de developpement | 软件的模块分解 | "代码如何组织成模块?" |
| **物理视图** | Vue de deploiement | 硬件上的物理分布 | "软件部署在哪些机器上?" |

这四视图加上**场景(Scenarios)**视角(用用例/场景串联所有视图),构成了著名的"4+1视图模型"。

## 3. MVC——最经典的交互式架构

### 3.1 MVC的三部分

> *Le patron Modele-Vue-Controleur, ou MVC, decompose une application en trois sous-parties.*

| 部分 | 法文 | 职责 | 不负责什么 |
|------|------|------|-----------|
| **模型** | Modele | 业务逻辑 + 数据访问 (logique metier, acces aux donnees) | 不处理UI |
| **视图** | Vue | 与用户的交互 (affichage, interactions utilisateur) | 不包含业务逻辑 |
| **控制器** | Controleur | 协调M和V,处理用户输入,更新模型和视图 | 不直接操作数据 |

### 3.2 MVC中的信息流

```
用户操作 -> 控制器 -> 更新模型 -> 模型通知视图 -> 视图刷新显示
```

关键原则: **模型不知道视图的存在**。模型通过观察者模式(Observer Pattern)通知视图,但不知道有多少视图、视图是什么样子。

### 3.3 MVC在Web应用中的变体

课件指出,PHP/Java Web框架中的MVC有细微变化:
- 模型 = 数据库访问 + 业务逻辑
- 视图 = HTML/CSS/JS 页面渲染
- 控制器 = 路由 + 请求处理 + 视图选择

## 4. 分层架构——关注点分离的垂直组织

### 4.1 核心思想

> *Organise le systeme en couches fournissant chacune un ensemble de services. Chaque couche ne peut communiquer qu'avec les couches adjacentes.*

分层架构将系统组织为提供服务的层次,每层只能与**相邻层**通信。

```
+------------------+
|    表现层        | <- UI
+------------------+
|    业务逻辑层    | <- 核心功能
+------------------+
|    数据访问层    | <- 持久化
+------------------+
|    数据库        | <- 存储
+------------------+
```

### 4.2 N层架构 (N-tier)

课件给出了典型的三层架构示例: **浏览器(表现层) -> Web服务器(业务层) -> 数据库(数据层)** = 3-tiers架构。

当每层对应一个**独立进程**运行在**不同机器**上时,称为n-tier架构。

### 4.3 优劣势

| 优势 | 劣势 |
|------|------|
| 支持**增量开发**(每层独立开发) | 层间通信有性能开销 |
| 层接口变更只影响相邻层 | 不是所有系统都有清晰的分层边界 |
| 底层服务可被全系统复用 | 严格的相邻通信规则可能导致冗余 |

## 5. 客户端-服务器架构——网络时代的基石

### 5.1 核心思想

系统被分解为:
- **服务器(Server)**: 提供服务(数据、计算、文件...)
- **客户端(Client)**: 请求服务并呈现结果

### 5.2 关键特征

- 分布式系统的基础架构
- 客户端和服务器通过网络通信
- 一个服务器可以服务多个客户端
- 典型应用: Web应用、数据库系统、文件共享

## 6. 数据中心架构(仓储)——以数据为中心

### 6.1 核心思想

> 所有共享数据集中在一个**中央数据库/仓储(Repository)**中,所有子系统通过它交换数据。

```
子系统A ----\
子系统B ----->> 中央仓储 (Repository) <<----- 子系统D
子系统C ----//
```

### 6.2 适用场景

- 数据驱动的系统(如ERP、CMS)
- 多个子系统需要**共享大量数据**
- 对数据一致性和完整性要求高的系统

## 7. 管道-过滤器架构——数据流的变换链

### 7.1 核心思想

> 系统被组织为一系列**数据处理组件(过滤器/Filters)**,通过**管道(Pipes)**连接,数据沿管道顺序流动并逐步变换。

```
输入 -> [Filter1] -> [Filter2] -> [Filter3] -> 输出
        (变换1)      (变换2)      (变换3)
```

### 7.2 经典示例

- **Unix命令行管道**: `cat file.txt | grep "error" | sort | uniq -c`
- **编译器**: 词法分析 -> 语法分析 -> 语义分析 -> 代码生成
- **ETL管道**: 数据提取 -> 清洗 -> 转换 -> 加载

### 7.3 优劣势

| 优势 | 劣势 |
|------|------|
| 易于理解(数据流直观) | 不适合交互式系统 |
| 过滤器可独立复用 | 数据格式需要在过滤器间约定 |
| 支持并行处理 | 错误处理复杂 |

## 五种架构模式速查表

| 模式 | 核心思想 | 何时使用 | 典型场景 |
|------|---------|---------|---------|
| **MVC** | 分离数据(M)、显示(V)、控制(C) | 交互式系统 | Web应用、桌面应用 |
| **分层** | 垂直分解为功能层次 | 需要增量开发,关注点分离 | 企业应用、OS |
| **客户端-服务器** | 服务提供方+请求方 | 分布式系统,网络环境 | Web、数据库 |
| **仓储** | 中央数据存储,子系统共享 | 数据密集型,多子系统共享 | ERP、CMS |
| **管道-过滤器** | 数据流经一系列变换器 | 批量数据处理,可组合变换 | 编译器、ETL |
