## Partie A: Gestion des Erreurs / 错误处理

## 1. 错误的系统性分类

### 1.1 完整分类树

课件构建了从根源到处理策略的完整错误分类体系:

```
错误 (Erreurs)
  |
  +-- 编译期错误 (Compilation)
  |     |
  |     +-- 词法错误 (Lexique): 拼写错误、不识别的符号
  |     +-- 语法错误 (Syntaxe): 不符合语言语法规则
  |     +-- 静态语义错误 (Semantique statique): 类型不匹配
  |           (编译器可检测)
  |
  +-- 运行期错误 (Execution)
        |
        +-- 动态语义错误 (Semantique dynamique)
        |     (运行时异常: 除零、空指针、越界)
        |     -> 通过测试和异常处理来管理
        |
        +-- 逻辑错误 (Logique)
              (程序运行但不产生正确结果)
              -> 通过测试、断言、代码审查来发现
```

### 1.2 各层级的发现策略

| 错误类型 | 发现时机 | 主要手段 |
|---------|---------|---------|
| 词法/语法/静态语义 | 编译时 | 编译器自动报告 |
| 动态语义 | 运行时 | 测试 + 异常处理机制 |
| 逻辑错误 | 运行时(结果不正确) | 断言 + 精心设计的测试用例 + 代码审查 |

## 2. 防御式编程——让错误尽早暴露

### 2.1 核心原则

防御式编程(Defensive Programming)的核心思想: **不要让错误悄悄传播**。一旦发现异常状态,立即报告,越早越好。

### 2.2 关键实践

| 实践 | 说明 |
|------|------|
| **前置条件检查** | 函数入口处验证参数合法性 |
| **后置条件检查** | 函数出口处验证返回值合理性 |
| **不变量检查** | 关键数据结构的一致性断言 |
| **Fail Fast** | 遇到不可恢复的错误立即终止,而非"试图继续" |

### 2.3 为什么"Fail Fast"比"试图继续"更好?

- 错误被隐藏 -> 可能在很久以后才暴露 -> 调试极其困难
- 错误立即暴露 -> 堆栈信息清晰 -> 根因分析简单
- 错误被"宽容"处理 -> 数据可能被损坏 -> 后续一切不确定

## 3. Java断言 (Assertions)——开发期的安全网

### 3.1 什么是断言?

> *Mot-cle utilise a des fins de debogage. Permet aux developpeurs de tester les hypotheses concernant leur code.*
> 用于调试目的的关键字。允许开发者测试关于代码的假设。

### 3.2 断言的特性

| 特性 | 说明 |
|------|------|
| **用途** | 测试逻辑错误 |
| **可开关** | 可在运行时启用(-ea)或禁用(-da) |
| **生产环境** | 通常禁用(避免性能开销) |
| **语法** | `assert condition : "message";` |

### 3.3 启用/禁用

```
# 命令行启用
java -ea Main

# Eclipse中启用
Run -> Run Configurations -> Arguments -> VM Arguments: -ea

# 禁用
java -da Main  (或留空)
```

## 4. 异常处理策略——捕获、传播还是恢复?

### 4.1 三种策略

| 策略 | 场景 | 示例 |
|------|------|------|
| **捕获并处理** | 可以合理恢复的错误 | 文件不存在 -> 使用默认配置 |
| **捕获、包装、传播** | 需要添加上下文信息 | SQL异常 -> 包装为"数据访问失败" |
| **不捕获,直接传播** | 调用者无法处理的错误 | 内存不足、不可恢复的状态损坏 |

### 4.2 异常的层次设计

课件暗示了异常体系的层次结构:
- **Checked Exceptions**(Java): 调用者**必须**处理(捕获或声明抛出)
- **Unchecked Exceptions**(Java): 调用者**可以**处理(通常是编程错误)

## Partie B: Cloture du Projet / 项目收尾

## 5. 为什么需要正式的项目收尾?

### 5.1 项目管理的最后一环

回顾第04章的项目管理五阶段,收尾(Cloture)是第五阶段也是**最容易被忽略的阶段**。项目"做完了"不等于"收尾了"。

### 5.2 收尾的核心目标

| 目标 | 如果不做会怎样 |
|------|--------------|
| **确认交付完整性** | 客户认为没完成,团队认为完成了 -> 纠纷 |
| **知识沉淀** | 同样的错误在下个项目中再次发生 |
| **团队解散** | 人员被"挂起",资源无法释放 |
| **正式签署** | 合同/法律风险 |

---

## 6. 收尾阶段的核心活动

课件将收尾分解为以下关键活动:

| 活动 | 内容 |
|------|------|
| **交付物验收** | 确认所有工作已完成、批准和移交 |
| **文档归档** | 所有项目文档(需求、设计、测试报告)存档 |
| **合同收尾** | 与客户/供应商的合同正式关闭 |
| **资源释放** | 人员重新分配、硬件归还、许可证回收 |
| **经验总结** | 记录成功和失败的经验教训 |

---

## 7. 经验教训——从成功和失败中学习

### 7.1 经验教训的内容

> *Etude des lecons apprises au cours du projet et faire le bilan des reussites et des echecs.*
> 研究项目中学到的教训,总结成功与失败。

| 应该记录 | 具体问题 |
|---------|---------|
| **什么做对了?** | 哪些实践、工具、流程促成了成功? |
| **什么做错了?** | 哪些决策导致了问题?如何避免? |
| **如果重来会怎么做?** | 具体的改进建议 |
| **意外发现** | 计划之外的收获 |

### 7.2 经验教训的反馈循环

```
项目A的教训 -> 记录 -> 可供项目B参考 -> 项目B避免重复错误 -> 记录新教训 -> ...
```

这形成了一个**组织级的学习循环**,是软件工程成熟度模型(CMMI)中高成熟度组织的核心特征。
