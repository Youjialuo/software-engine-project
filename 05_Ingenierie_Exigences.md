## 1. 什么是需求?——从模糊意图到精确规格

### 1.1 需求的定义

> 课件原文: *Les exigences d'un systeme sont les descriptions de ce que le systeme doit faire - les services qu'il fournit et les contraintes sous lesquelles il s'executera et sera developpe.*
> 系统需求是对系统必须做什么的描述——它提供的服务和它在运行和开发中所受的约束。

关键要点:
- 需求反映客户对服务于某个特定目的(控制设备、下订单、查找信息)的系统的**需要**
- 模糊的需求会导致开发者和用户之间产生**不同的解释**
- 原则上,需求应该**完整**(Completes)和**一致**(Consistantes)

### 1.2 目标 vs 需求——最关键的概念辨析

课件非常清晰地划分了两个层次:

| | 目标 (Objectif) | 需求 (Exigence) |
|------|---------------|----------------|
| **本质** | 用户的意图 (intention generale) | 系统必须做什么的精确描述 |
| **可测性** | 不可客观测试 | 可使用可客观测试的度量 |
| **示例** | "易于使用" | "系统所需学习时间不超过4小时" |
| **作用** | 回答"为什么"和"是什么" | 回答"怎么做" |

**深层解读**: 这个区分是需求工程中最易出错的地方。如果把"目标"当作"需求",就会出问题——"易于使用"作为一个需求是无法验证的(什么叫"易于"?对谁而言?)。而"学习时间不超过4小时"是可以**客观测试**的。

## 2. 需求的分类体系——功能性、非功能性、领域需求

### 2.1 三级分类

课件构建了一个层次化的需求分类体系:

```
用户需求 (Exigences Utilisateur)
    - 定义产品的最终目的
    - 回答"是什么"(Quoi)和"为什么"(Pourquoi)的问题

系统需求 (Exigences Systeme)
    - 软件必须拥有的功能特性
    - 回答"怎么做"(Comment)的问题

领域需求 (Exigences de Domaine)
    - 系统在其使用环境中的约束
```

### 2.2 功能性需求 vs 非功能性需求 (Systeme层级)

这是考试和应用中最核心的分类:

| 功能性需求 (Fonctionnelles) | 非功能性需求 (Non-Fonctionnelles) |
|---------------------------|--------------------------------|
| 系统应提供的**服务**(做什么) | 对这些服务的**约束**(做得如何) |
| 如何对各种**输入**做出反应 | **时间**约束 |
| 在各种**情况**下的行为 | **开发过程**约束 |
| 也可以声明系统**不应**做什么 | **标准**的使用 |
| 通常针对特定功能 | 通常适用于系统**整体** |

**判断口诀**: 功能性需求回答"系统做什么"(功能),非功能性需求回答"系统做得怎么样"(质量属性)。

## 3. 非功能性需求的子类——性能、可用性、安全性等

课件将非功能性需求进一步细分为三个子类和多个维度:

### 3.1 三个子类

| 子类 | 法文 | 来源 |
|------|------|------|
| **产品需求** | Exigences de produit | 指定产品应如何表现(响应时间、可靠性等) |
| **组织需求** | Exigences organisationnelles | 来自组织策略和程序(标准、流程等) |
| **外部需求** | Exigences externes | 来自系统外部的因素(法律、伦理等) |

### 3.2 产品需求的8个维度

| 维度 | 法文 | 关键问题 |
|------|------|---------|
| **性能** | Performance | 响应速度有多快?吞吐量多大? |
| **可靠性** | Fiabilite | 计算和精度的可靠性? |
| **存储** | Taille de stockage | 占用多少内存、CPU、存储? |
| **可用性** | Utilisabilite | 需要多少努力来学习、使用? |
| **安全性** | Securite | 如何防御未授权访问? |
| **鲁棒性** | Robustesse | 面对无效数据时的表现? |
| **适应性** | Adaptabilite | 能否适应其他环境或问题? |

## 4. 需求工程过程——从获取到管理的完整链条

### 4.1 在软件过程中的位置

课件将"需求规格说明"定位为软件过程的**四大基本活动之首**:

```
1. Specification des exigences (需求规格说明)
2. Conception (设计)
3. Implementation (实现)
4. Validation (验证)
```

### 4.2 需求工程的核心活动

| 活动 | 法文 | 核心问题 |
|------|------|---------|
| **获取** | Elicitation | 如何发现用户真正需要什么? |
| **分析** | Analyse | 这些需求是否一致、完整、可实现? |
| **规格说明** | Specification | 如何把需求准确地写下来? |
| **验证** | Validation | 需求是否正确反映了用户的需要? |
| **管理** | Gestion | 需求变更时如何控制? |

### 4.3 利益相关者 (Parties Prenantes)

需求工程涉及多方利益相关者:
- **客户**: 出资方,有时也是最终用户
- **最终用户**: 实际使用系统的人
- **领域专家**: 了解业务规则的人
- **监管者**: 如法律/合规部门
- **开发者**: 实现系统的人

## 5. DSEL/SRS——软件需求规格说明文档

### 5.1 定义

> *Le document de specification des exigences (DSEL/SRS) est l'enonce officiel de ce qui est attendu de la part des developpeurs du systeme.*

软件需求规格说明文档是对开发人员预期的正式声明。它应该同时包含:
- **用户需求** (Exigences utilisateur)
- **系统需求** (Exigences systeme)
- **规格说明** (Specifications)

### 5.2 关键原则

课件特别强调: **这不是设计文档** (Ce n'est pas un document de conception)。它关注的是 **"是什么"(QUOI)** 而非 **"怎么做"(COMMENT)**。

### 5.3 敏捷环境下的角色

课件指出: 在敏捷开发中,SRS的作用较小。因为敏捷方法通过**频繁的用户反馈**来替代一次性的详细需求文档。

## 6. 需求规格的编写方式——自然语言与结构化语言

### 6.1 自然语言 (Langage naturel)

使用编号的自然语言句子,每句表达一个需求。

| 优势 | 劣势 |
|------|------|
| 表达力强、直观、通用 | **缺乏清晰度**: 精确性难以保证而不使文档难以阅读 |
| 用户和客户都能理解 | **需求混淆**: 功能性和非功能性需求容易混杂 |
| | **需求合并**: 多个不同需求可能被合并表达 |

### 6.2 结构化自然语言 (Langage naturel structure)

使用标准表单或模板,每个字段提供需求某个方面的信息。

| 优势 | 劣势 |
|------|------|
| 标准化编写 | 编写者的自由度受限 |
| 清晰 | |
| 易于编写 | |
| 可封装 | |

### 6.3 结构化语言的实例模板

课件通过胰岛素泵案例展示了结构化规格模板:

| 字段 | 内容 |
|------|------|
| **功能** (Fonction) | 计算胰岛素剂量 |
| **描述** (Description) | 当血糖在安全区(3-7单位)时计算注射剂量 |
| **输入** (Entree) | 当前血糖(r2),前两次读数(r0, r1) |
| **来源** (Source) | 当前读数来自传感器,历史读数来自内存 |
| **输出** (Sortie) | CompDose——要注射的胰岛素剂量 |
| **目标** (Destination) | 主控制循环 |
| **动作** (Action) | 详细计算逻辑(条件判断+公式) |
| **约束** (Contrainte) | 需要两次历史读数才能计算变化率 |
| **前置条件** (Precondition) | 胰岛素储备 >= 最大允许单次剂量 |
| **后置条件** (Postcondition) | r0 <= r1; r1 <= r2 |
| **副作用** (Effet de bord) | 无 |

## 7. 案例研究——胰岛素泵系统

### 7.1 系统背景

个人胰岛素泵是一个**嵌入式系统**,用于糖尿病患者管理血糖水平。这是一个**安全关键系统**(systeme critique):
- 低血糖 -> 脑功能障碍、昏迷、死亡
- 高血糖 -> 眼睛、肾脏等严重损害

### 7.2 系统功能

- 从血糖传感器收集数据
- 基于血糖变化率计算胰岛素注射量
- 向微型泵发送信号以注射正确剂量
- 每10分钟测量一次
- 每分钟进行自检

### 7.3 需求示例

**自然语言需求**:
> "系统必须每10分钟测量血糖,并在需要时触发胰岛素注射。(血糖水平变化相对缓慢,因此更频繁的测量是不必要的;而不太频繁的测量可能导致血糖水平不必要地升高)"

> "系统必须每分钟启动自动自检程序。要测试的条件和相关动作在表X中定义。(自检程序可以发现硬件和软件问题,并警告用户某些操作可能无法执行)"

## 本章知识图谱

```
          需求 (Exigence) != 目标 (Objectif)
                    |
        +-----------+-----------+
        |           |           |
    用户需求      系统需求      领域需求
    (是什么)      (怎么做)      (使用环境)
                    |
        +-----------+-----------+
        |                       |
   功能性需求               非功能性需求
   (做什么)                 (做得如何)
                                |
              +-----------+-----+------+
              |           |            |
           产品需求    组织需求      外部需求
         (性能/可靠性  (标准/流程)   (法律/伦理)
          /可用性...)
```
