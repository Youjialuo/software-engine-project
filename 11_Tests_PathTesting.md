## Partie A: Les Tests en General / 软件测试总论

## 1. 测试的本质——寻找缺陷,而非证明正确

### 1.1 核心定义

> *Tester != prouver que le logiciel est exempt d'erreurs. Tester = trouver les defauts.*
> 测试 **不等于** 证明软件没有错误。测试 = **找到缺陷**。

这一定义有深远的哲学含义: 测试永远不能证明软件是正确的(因为输入空间无限),只能证明在某些输入下没有发现错误。这是Edsger Dijkstra的名言的根本:

> "Program testing can be used to show the presence of bugs, but never to show their absence."

### 1.2 测试的三大任务

1. **识别缺陷** (Identifier defauts): 找到bug、缺失的需求、不正确的功能、语义和逻辑错误
2. **耗时工作**: 无法测试所有组合和场景 -> 必须聚焦最关键的领域
3. **持续活动**: 测试不是"写完代码后做一次"的活动,而是贯穿整个开发过程

## 2. 错误分类——从编译错误到逻辑错误

### 2.1 错误的三大层级

课件以Java为例构建了完整的错误分类体系:

```
错误 (Erreurs)
  |
  +-- 编译错误 (Compilation)
  |     |
  |     +-- 词法错误 (Lexique): 拼写错误、无效字符
  |     +-- 语法错误 (Syntaxe): 括号不匹配、缺少分号
  |     +-- 静态语义错误 (Semantique statique): 类型不匹配
  |
  +-- 执行错误 (Execution)
        |
        +-- 动态语义错误 (Semantique dynamique): 除零、空指针
        +-- 逻辑错误 (Logique): 算法正确但结果错误
```

### 2.2 如何发现这些错误?

| 错误类型 | 发现方式 |
|---------|---------|
| 编译错误 | 编译器直接报告 |
| 动态语义错误 | 运行时异常;通过测试发现 |
| 逻辑错误 | 最隐蔽;需要精心设计的**测试用例**和**代码审查** |

## 3. 测试金字塔——单元、集成、系统、验收

### 3.1 经典测试金字塔

```
       /\
      /验收\        <- 少量,验证业务需求
     /------\
    / 系统测试 \     <- 端到端功能验证
   /----------\
  /  集成测试    \   <- 模块间接口验证
 /--------------\
/    单元测试      \ <- 大量,每个函数/类
+------------------+
```

### 3.2 各级测试详解

| 级别 | 测试什么 | 谁做 | 工具 | 速度 |
|------|---------|------|------|------|
| **单元测试** | 单个函数/方法 | 开发者 | JUnit, pytest | 极快(毫秒) |
| **集成测试** | 模块间的接口 | 开发者/测试工程师 | 测试框架 | 快(秒) |
| **系统测试** | 整个系统的功能 | 独立测试团队 | 自动化+手动 | 慢(分钟-小时) |
| **验收测试** | 是否满足用户需求 | 用户/客户代表 | 手动/BDD工具 | 取决于场景 |

## 4. 功能性测试 vs 非功能性测试

课件基于需求的二元分类对应划分了测试类型:

### 4.1 功能性测试

> 验证系统**做了什么**(Quoi)和**怎么做**(Comment)。

包括: 单元测试、集成测试、系统测试、验收测试。

### 4.2 非功能性测试

> 验证系统的**质量属性**。

| 测试类型 | 关注点 | 典型问题 |
|---------|--------|---------|
| **性能测试** | 响应时间、吞吐量 | "1000用户并发时系统多快?" |
| **安全测试** | 未授权访问防护 | "能否绕过认证?" |
| **可用性测试** | 易学易用 | "用户能在5分钟内完成首次操作吗?" |
| **兼容性测试** | 跨平台/跨浏览器 | "在IE和Chrome上都能运行吗?" |
| **可靠性测试** | 长期稳定运行 | "连续运行7天会崩溃吗?" |

## Partie B: Path Testing / 路径测试

## 6. 白盒测试与路径测试基础

### 6.1 什么是路径测试?

> *Le test de chemin est une approche de test qui consiste a s'assurer que chaque chemin d'execution d'un programme a ete execute au moins une fois.*
> 路径测试确保程序的**每条执行路径**至少被执行一次。

### 6.2 为什么是白盒测试?

- 由**开发团队**在代码编写期间执行
- 需要**看见程序内部结构**(因此是白盒/结构测试)
- 属于**开发测试**(Tests de developpement)而非用户验收测试

---

## 7. 控制流图 (CFG) 的构建

### 7.1 从代码到CFG

课件用以下示例展示:

```
代码:
1. if (X > 0) then           CFG:
2.    if (Y > Z) then          [1] --> [2] --> [3]
3.       X = Y                 |       |
4.    else X = Z               |       +-----> [4]
5. print X                     +----------------> [5]
                               [5]
```

### 7.2 CFG元素

| 元素 | 含义 |
|------|------|
| **节点 (Noeud)** | 一个基本代码块(顺序执行) |
| **谓词节点 (Predicat)** | 包含条件判断的节点(出度 >= 2) |
| **边 (Arc)** | 程序流向 |

## 8. 圈复杂度 (Cyclomatic Complexity) 的计算

### 8.1 定义

> *La complexite cyclomatique (cc) mesure le nombre de chemins lineairement independants dans un programme. Plus cc est eleve, plus le code est complexe.*
> 圈复杂度衡量程序中**线性独立路径**的数量。cc越高,代码越复杂。

### 8.2 两种计算方法

课件给出两种等价方法:

| 方法 | 公式 | 示例(上图CFG) |
|------|------|-------------|
| **边-节点法** | cc = arcs - noeuds + 2 | 6 - 5 + 2 = 3 |
| **谓词节点法** | cc = 谓词节点数 + 1 | 2 + 1 = 3 |

### 8.3 圈复杂度的工程意义

- **确定测试路径数**: cc = 需要的最少测试路径数
- **衡量代码复杂性**: cc > 10 通常被视为"需要重构"
- **风险评估**: cc越高,缺陷密度越高

## 9. 从圈复杂度到测试用例生成

### 9.1 四步测试法

课件给出了完整路径:

```
Step 1: 编写代码
Step 2: 画CFG
Step 3: 计算cc
Step 4: 找出一组路径(cc条)
Step 5: 为每条路径生成测试用例
```

### 9.2 示例

对上述代码(cc=3),三条独立路径:

| 路径 | 测试用例 | 覆盖场景 |
|------|---------|---------|
| `1, 2, 3, 5` | X=1, Y=2, Z=1 | X>0且Y>Z |
| `1, 2, 4, 5` | X=1, Y=1, Z=2 | X>0但Y<=Z |
| `1, 5` | X=0, Y=任意, Z=任意 | X<=0(跳过内层) |

**关键**: 每个路径对应一个测试用例。cc=3意味着至少需要3个测试用例才能覆盖所有独立路径。
